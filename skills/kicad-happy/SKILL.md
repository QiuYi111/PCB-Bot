# KiCad Happy Design Review

Coordinate reliable KiCad analysis and manufacturing-readiness review. This Skill is a project-level wrapper around the existing `kicad` Skill and its analyzers; it owns workflow, evidence, configuration, and handoffs, not a second S-expression parser.

**REQUIRED SUB-SKILL:** Use `kicad` for KiCad parsing, structured analyzer output, raw-file fallbacks, and deep-review methodology.

## Toolchain preflight

Before automated review, follow the installation and configuration chapter in the `pcb` Skill and run `scripts/check_toolchain.py`. KiCad CLI is required for native DRC/ERC and export gates; ngspice, datasheets, supplier APIs, and `gh` are optional capabilities whose absence must be recorded explicitly. Do not replace a missing native check with a manual claim.

## Review workflow

### 1. Inventory the project

Scan for all applicable inputs before choosing analyzers:

- `.kicad_sch`, `.kicad_pcb`, `.kicad_pro`, `.kicad_dru`;
- legacy `.sch`, `.lib`, `.net`, `.xml`;
- Gerber/Excellon directories;
- `datasheets/`, BOMs, prior reports, `analysis/`, and `.kicad-happy.json`.

If the project was just generated by `circuit-synth`, also read the source Python and the generation log.

### 2. Resolve configuration and evidence

Use `.kicad-happy.json` when present. Respect its project metadata, design intent, power-rail overrides, BOM conventions, suppressions, analysis output directory, retention, and auto-diff settings.

Before making pin-level or electrical claims, look for manufacturer datasheets or structured extractions. KiCad library agreement is consistency evidence, not physical correctness evidence. If datasheets are missing, state that the affected claims are unverified or consistency-only.

### 3. Run applicable analyzers

When a project-local `pcbflow` environment is present, run its `preflight` and `review --full` commands before invoking analyzers manually. Consume the resulting manifest and gate report as the canonical run record; use manual analyzer commands only to debug a failed gate or to run a narrower follow-up. Never infer native DRC success from `kicad-cli` availability alone: require a successful command and report file.

Use the existing `kicad` scripts and keep outputs in the configured `analysis/` run directory when possible:

```bash
python3 <kicad-skill-path>/scripts/analyze_schematic.py design.kicad_sch --analysis-dir analysis/
python3 <kicad-skill-path>/scripts/analyze_pcb.py design.kicad_pcb --full --analysis-dir analysis/
python3 <kicad-skill-path>/scripts/cross_analysis.py \
  --schematic analysis/<run>/schematic.json \
  --pcb analysis/<run>/pcb.json \
  --analysis-dir analysis/
python3 <kicad-skill-path>/scripts/analyze_gerbers.py gerbers/ --analysis-dir analysis/
```

Run only the commands supported by the files present, but disclose every skipped or failed applicable check. For a full review, also consider the existing `emc`, `spice`, thermal, lifecycle, datasheet, BOM, and fabrication Skills.

### 4. Verify findings

Treat analyzer JSON as structured evidence, not the final verdict:

- confirm component counts against raw files;
- verify critical pin-to-net mappings against raw schematic and datasheet;
- cross-check schematic pins against PCB pads and footprints;
- inspect power rails, decoupling, protection, connectors, clocks, buses, and test points;
- check whether copper zones were filled before trusting copper-presence conclusions;
- distinguish deterministic, heuristic, topology-based, and datasheet-backed findings;
- triage likely false positives before labeling a blocker.

For motor/power boards, require an explicit load model in the project context (for example `pcbflow.json`) and check estimated rail current against the actual copper path or pour. Do not treat a small-signal or unloaded thermal result as a motor-load validation.

For generated designs, compare the requirements/topology contract from `circuit-synth` against the generated nets, components, values, footprints, and project settings.

### 5. Produce a review result

Use a concise but complete report with:

1. verdict and scope;
2. files and analyzers run;
3. component/BOM and power-tree summary;
4. schematic findings;
5. PCB, cross-domain, DRC/ERC, DFM, EMC, thermal, SPICE, and Gerber findings as applicable;
6. prioritized blockers and recommended fixes;
7. evidence basis and confidence;
8. skipped analyses, missing datasheets, and residual risks;
9. next handoff or release decision.

Do not call a design “ready to manufacture” when a required analysis failed, a critical datasheet is missing, or schematic/PCB synchronization is unresolved.

## Handoffs

| Need | Handoff |
|---|---|
| Change topology or generate new circuit code | `circuit-synth` |
| Extract/enrich/export BOM | `bom` |
| Manufacturer pinout and required application circuit | `datasheets` or supplier Skill |
| SPICE simulation, parasitics, or value sweep | `spice` |
| EMC pre-compliance | `emc` |
| LCSC/JLCPCB production and DFM | `lcsc` / `jlcpcb` |
| Alternative fabrication | `pcbway` |

Pass the analysis directory, current run ID, source files, findings, and verification gaps to every downstream handoff. Never silently replace a failed handoff with an unsupported claim.

## Safety and scope

Analysis is read-only by default. Do not overwrite source KiCad files, refill zones, change design rules, or suppress findings without explicit authorization. A suppression must retain a reason and remain visible in the report.

If the user asks for a topology fix, stop reviewing and route the requested design change to `circuit-synth`; after regeneration, return here for regression analysis.
